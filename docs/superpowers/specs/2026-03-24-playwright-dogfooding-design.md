# Playwright Dogfooding Design

## Overview

Playwright MCPを使用してGovAssistの全機能をインタラクティブにドッグフーディングし、機能や仕様の不備を発見する。

## Scope

- 対象: 実装済みの全機能（校正、履歴CRUD、DOCXエクスポート、設定、モデル選択）
- AI API: 本物のSAKURA AI Engine APIを使用（APIキー必須）
- テストデータ: 複数パターン（短文、長文、記号混じり、エッジケース）
- 方式: シナリオ順次実行（体系的な網羅）

## Prerequisites

- Frontend dev server: `npm run dev` (port 5173)
- Backend server: `uvicorn main:app --reload` (port 8000)
- `.env` に `AI_ENGINE_API_KEY` が設定済み

## Testing Notes

- シナリオ1-4は順次依存（アプリ状態が蓄積される）
- シナリオ5はシナリオ2-3で履歴を保存済みであることが前提
- AI応答は非決定的なので、結果の内容よりUIの振る舞いに注目
- スクリーンショットは各シナリオ完了時とエラー発生時に取得
- design specとの乖離があれば「既知の逸脱」として記録する

## Test Data

| ID | パターン | 内容 |
|----|---------|------|
| ① | 短いメール | 敬語ミス、誤字を含む3-5行のビジネスメール |
| ② | 長めの文書 | 2000文字程度の報告書（段落構成あり） |
| ③ | 記号・英数字混じり | URL、数値、記号が混在する通知文 |
| ④ | エッジケース用 | 各種エッジケース用の短いテキスト |

## Test Scenarios

### Scenario 1: Startup & Initial Display

- `http://localhost:5173` を開く
- BootSequenceアニメーションの動作確認
- WarningModalの表示→閉じる
- 初期画面のレイアウト確認（Header、SideMenu、メインエリア）
- MatrixRain背景の表示
- Headerのモデルセレクターの表示確認

### Scenario 2: Proofreading — Basic Flow

- テストデータ①（短いメール）を入力
- 文書種別「メール」を選択
- オプションをデフォルトで実行
- 結果の3タブ（「校正前」「校正後」「差分」）を確認
- コピー機能の動作確認
- DOCXダウンロードの動作確認
- 履歴保存の動作確認
- ※ design specではタブ1=ハイライト表示、タブ2=比較表示を想定しているが、現在の実装は「校正前」「校正後」が全文テキスト、「差分」が修正一覧。この乖離を記録する

### Scenario 3: Proofreading — Multiple Patterns & Status Branching

- テストデータ②（長めの文書）で校正
- テストデータ③（記号・英数字混じり）で校正
- オプション変更時の動作確認
- リトライ機能の確認
- ※ `large_rewrite`警告が表示されるケースがあれば確認（AIが大幅な修正を行った場合）
- ※ `partial`/`error`ステータスになった場合、UIの分岐が正しいか確認

### Scenario 4: File Upload

- .docxファイルのアップロード（ボタン操作）
- .docxファイルのドラッグ&ドロップ
- .pdfファイルのアップロード
- ファイル抽出後の抽出元バナー表示確認
- 抽出テキストの編集可能確認
- 「元に戻す」ボタンの動作確認
- 抽出後に自動送信されないことの確認
- 不正ファイルのリジェクト確認

### Scenario 5: History Management

- 履歴一覧の表示（シナリオ2-3で保存した履歴が表示されるか）
- 履歴項目クリックで詳細表示（HistoryDetail）
  - 入力テキストと結果の表示確認
  - メモ編集→保存→永続性確認
  - 戻るボタンで一覧に戻る
- 検索機能（キーワード検索）
- フィルタ（文書種別、日付範囲）
- ページネーション（20件/ページ）
- 個別削除
- 全削除（確認ダイアログの確認）
- ※ 切断レコード（100KB超過）の警告表示があれば確認

### Scenario 6: Settings & Persistence

- AIモデルの切替
- デフォルト文書種別の変更
- デフォルトオプションの変更
- サーバー設定（履歴上限）の保存
  - 境界値テスト: 0（拒否）、1（受理）、200（受理）、201（拒否）、非数値（拒否）
- ページリロード後の設定永続性確認
  - Settingsで変更したモデルがHeaderのセレクターに反映されるか
  - デフォルト文書種別が校正画面に反映されるか
- localStorageキー `govassist_settings` の内容確認

### Scenario 7: Edge Cases

- 空入力で送信ボタンの状態確認
- 8000文字ギリギリの入力
- 非常に短いテキスト（1文字）
- 特殊文字（絵文字、記号のみ）
- 連続改行（3+）の正規化確認
- 行頭・行末空白のトリム確認
- 文字カウンターの動作確認

### Scenario 8: UI/UX & Navigation

- SideMenuのナビゲーション
  - 校正ツール、履歴（有効）
  - 翻訳ツール、要約ツール、フォーマット（無効化）
  - ※ design specではサイドメニューに「履歴」は含まれていない。またPhase2/3のプレースホルダー名が異なる（実装: 翻訳/要約/フォーマット、spec: 文書要約・翻訳/PDF加工/AIチャット）
- 無効化メニュー項目のクリック時の振る舞い
- リサイズ時のレイアウト崩れ確認
- 各種アニメーションの動作確認

### Scenario 9: Error Handling & Resilience

- ※ AIタイムアウトやレートリミットは意図的に発生させにくいため、自然なテスト中に発生した場合は確認
- `error` ステータス時のリトライボタン表示確認
- `validation_error`時のリトライボタン非表示確認
- ネットワークエラー時の表示確認（バックエンド停止状態でテスト可能）

## Deliverables

- 発見した不具合・仕様不備のリスト（design spec乖離を含む）
- スクリーンショット付きレポート
